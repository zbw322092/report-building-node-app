# CUSTOMIZATION
One principle we should follow is that, if native method is avaliable and fulfill our requirements perfectly, we should use it without any hesitation since native mathod has **_unmatchable proformance_** than the script snippets we written.
DO NOT use custom scrollbars unless you know what you are doing.

Customizing scrollbars is always a controversial issue. On the one hand, it provides full control of scrollbars. But on the other hand, it degrades user experience because native behavior is unmatchable. As the author of this script, I don't really want you to use it unless you are sure about what you are doing.

-- Dolphin Wood, the author of smooth-scrollbar
However...
In lots of situation, we really need customized scroll behaviors that browser cannot provided natively.
For examples:
window.scrollTo() is great, but if we would like "scroll to" behaviors happen within a specified container which is not the window object. window.scrollTo() cannot not help us.
Element.scrollIntoView() works well, until we need element showing within viewport with some customized offsets.
scroll-behavior: smooth is elegant, but we cannot customize detailed scroll behavior -- maybe we would like elements scroll quicker in first half distance, with slower scrolling speed during second half distance.
Also, scroll bar "momentum" scrolling style provided by -webkit-overflow-scrolling: touch is monotonous. If we want longer momentum slow down distance, for example, we need the help of external scripts.
